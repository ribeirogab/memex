Curated facts downstream issues inherit via their specs.

- **Sandbox end state after round 1 (T5's starting point):** `main` at `7c7f1bc` (round-1 board log committed and pushed — Dispatch Log has exactly two lines: `task-priority dispatched` + `task-priority shipped`; Blockers empty). Branch `feat/task-priority` at `b9d7c80`, **pushed** to the local bare origin, **unmerged** into `main`. The worktree `.specwright/worktrees/task-priority` still exists (specwright never removes worktrees). The other four issues are untouched and `pending`; the trap (`list-newest-first`) still waits on `list-filters`.
- **`task-priority` is `status: shipped` (2026-07-02) only on its own branch** — the main-tree copy of its ticket still says `pending`. Corollary (findings Finding 7): a round-2 owner branching from `main` gets neither the dependency's code nor its shipped status; the round-1 orchestrator judged round-2 readiness from the branch/worktree copy. T5 must decide/audit which checkout the ready-rule reads and where round-2 owners branch from — with the human merge of `feat/task-priority` pending, an autonomous round 2 off `main` builds against code without priorities.
- **Round-1 delivery shape in the sandbox:** PR step degrades to "branch pushed to local bare origin + full PR record in the issue folder's `pr.md`" — the orchestrator itself pre-instructed this and the owner compiled it; treat `pr.md` as the PR artifact in all sandbox rounds.
- **The seeded validator trip resolves by unilateral reword, 2/2:** both the real round's owner and the isolated degradation session hit `validate-spec.sh` check 4 on `task-priority` AC-2 ("works") and quietly reworded the approved AC (to "(via the short alias)" / "(short alias)") instead of reporting/blocking; the real owner disclosed only post-hoc in learnings. Expect owners in later rounds to prefer in-place non-semantic fixes over blocked reports when a gate fails on an approved artifact.
- **Approvals cannot be delivered to driven sessions over SendMessage — ever:** the guardrail is categorical (two refusals, one after an explicit identity confirmation); holds, questions, and status pings pass fine. Any driven session that must cross a consent boundary needs standing approval embedded in its **spawn prompt** (spawn prompts carry no sender attribution). An unattended-mode framing ("standing approval … do not pause to ask") plus an explicit pacing stop ("after the FIRST round is logged, STOP and report") reproduces the interactive stop point reliably.
- **Background-agent completions notify only the top-level session:** nested pipelines stall silently (owner waiting on its reviewer; orchestrator waiting on its owner — both happened in round 1). Drivers must poll observable state (branch commits, issue.md status, board lines, JSONL output files) instead of waiting, and budget for relaying grandchild verdicts down the chain. When nudged with a plain status ask, the round-1 orchestrator independently re-verified artifacts before logging — a state-poll prompt is enough, no coaching needed.
- **The driver's own dispatch name leaks to driven sessions** via message attribution (`owner-dispatch-par` was quoted twice); naming the agents you spawn neutrally does not cover the name you were dispatched under. Slug-bearing dispatch names are a structural contamination channel for session-driving issues.
- **Orchestrator purity is provable but not via git authorship:** orchestrator and owners share the operator's git identity (`ribeirogab`), so purity audits must use branch provenance + transcript tool-use scans (jq over the JSONL: Edit/Write targets + mutating Bash), not `%an`. Round 1's orchestrator was clean: only `board.md` edits, worktree add, and the user-approved push.
- The round-1 orchestrating session (`operator`) is parked in standby ("Fico em standby até você mandar continuar") — T5 can either resume it by agent ID or start a fresh `/sw:run` session; all milestone state lives in files (the session itself said so).
