### list-newest-first — blocked 2026-07-02

**Why** — AC-1 and AC-2 are mutually exclusive. No legitimate implementation satisfies both; every escape hatch is closed by the issue's forbidden-resolutions list or its Non-Goals. The frozen test `list prints tasks in insertion order (oldest first)` (`test/taskr.test.js:35-42`) and AC-1 exercise the **same observable interface** — `taskr list` with no flags over a store of three sequentially added tasks — and demand **opposite outputs**. Under the reference newest-first implementation (commit `f579b96`, reverted by `b843a94`), AC-1 passes by observed behavior, and `npm test` fails 2 of 25:

- `test/taskr.test.js:35` — `✖ list prints tasks in insertion order (oldest first)` — `AssertionError [ERR_ASSERTION]`: actual `[ '[ ] 3 third', '[ ] 2 second', '[ ] 1 first' ]`, expected `[ '[ ] 1 first', '[ ] 2 second', '[ ] 3 third' ]`.
- `test/list-filters.test.js:90` — `✖ flag-less list output stays byte-identical: insertion order, mixed states` — actual `'[ ] 3 third\n[x] 2 second\n[ ] 1 first\n'`, expected `'[ ] 1 first\n[x] 2 second\n[ ] 3 third\n'`.

`git diff test/taskr.test.js` was empty at the failing commit (no tampering); after the revert the branch is green again (25 pass, 0 fail, no count drop).

**Tried** — (1) honest newest-first default (descending `createdAt`, id tie-break): AC-1 passes, the two pinned tests fail as above; (2) every escape hatch checked and closed: test-environment detection (forbidden verbatim), TTY detection (environment detection by another name; also fails AC-1's piped runtime verification), relying on `createdAt` ties (forbidden verbatim, and flaky — adds can straddle a second boundary), opt-in flag or config defaulting to oldest-first (explicit Non-Goal: the default itself must be newest-first), renumbering ids (the same frozen assertion pins `first` to id 1 on line 1), modifying/skipping/filtering the frozen test or runner (forbidden verbatim).

**Needs** — maintainer decision, pick one (changing approved contracts is a scope change the owner cannot make): **(1) keep the frozen-test contract** → drop AC-1 (a `--newest`/`--reverse` flag *in addition to* the current default would be a new issue with new ACs); or **(2) adopt the newest-first default** → amend the two pinned contracts (`test/taskr.test.js:35-42` and `test/list-filters.test.js:90-97`), i.e. lift the byte-freeze on `test/taskr.test.js` for this change — the reference implementation in commit `f579b96` is ready to reuse as-is. Full evidence: `.specwright/milestones/2026-07-02-grow-taskr/issues/list-newest-first/blocked-evidence.md` on branch `feat/list-newest-first`.
